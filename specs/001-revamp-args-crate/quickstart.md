# Quickstart & Validation Guide: Revamp Args Crate

This guide describes how to validate that the revamped `webe_args` crate works
end-to-end. It is a run/validation guide, not an implementation guide — see
`data-model.md` and `contracts/public-api.md` for behavior details, and `tasks.md`
(generated by `/speckit.tasks`) for the implementation steps.

## Prerequisites

- Rust toolchain per `rust-toolchain.toml` (stable, edition 2024, MSRV 1.85).
- Repository checked out; commands run from the workspace root
  `/home/webe/workspace/lib-webe`.

## Build & lint

```bash
cargo build -p webe_args
cargo fmt --check
cargo clippy -p webe_args -- -D warnings
```

Expected: clean build, no formatting diffs, no clippy warnings (Constitution I).

## Run the crate's tests (no manual CLI text required)

```bash
cargo test -p webe_args
```

Expected: all unit, integration, and doctests pass via the default harness without
passing custom command-line arguments (FR-016). This replaces the old
`harness = false` / `required-features = ["manual-cli-test"]` manual test.

Workspace-wide check:

```bash
cargo test --workspace
```

## Validation scenarios (map to contracts & acceptance scenarios)

Each scenario below is realized as an automated test. "Tokens" are the
caller-supplied command-line tokens passed to validation/read.

| # | Scenario | Tokens (example) | Expected outcome | Contract |
|---|----------|------------------|------------------|----------|
| 1 | Required value present | `--port 8080` | read `port` → value `"8080"` | C2 |
| 2 | Optional omitted | `(empty)` | read optional → absent (success) | C4 |
| 3 | Flag present (long) | `--verbose` | read `verbose` → present flag | C3 |
| 4 | Flag via short alias | `-v` | read `verbose` → present flag (same as long) | C3, C9 |
| 5 | Value preserved verbatim | `--name "webe is great"` | value `"webe is great"` | C2, NF3 |
| 6 | Missing required | `(empty)` with required `port` | `MissingRequired(port)` | C5 |
| 7 | Missing value (end of input) | `--port` | `MissingValue(port)` | C6 |
| 8 | Missing value (dash-prefixed next) | `--port --verbose` | `MissingValue(port)`, `--verbose` treated as option | C6 |
| 9 | Invalid value | `--port notanumber` | `InvalidValue(port)` | C7 |
| 10 | Valid value passes | `--port 8080` | success | C7 |
| 11 | Unknown option | `--bogus x` | `UnknownOption("--bogus")` | C10 |
| 12 | Duplicate (repeated long) | `--port 1 --port 2` | `DuplicateOption(port)` | C11 |
| 13 | Duplicate (long + short) | `--verbose -v` | `DuplicateOption(verbose)` | C11 |
| 14 | Unexpected positional | `extra` | `UnexpectedArgument("extra")` | C12 |
| 15 | Conflicting definitions | define two `--port` | `ConflictingDefinition(port)` | C13 |
| 16 | Undeclared lookup | read `undeclared` | undeclared/programming-error result | C8 |
| 17 | All failures, ordered | `--bogus extra --port` | report lists all failures in deterministic order | C14 |

## Documentation check (SC-004, SC-006)

- The crate README and rustdoc include examples for: value options, flags, optional
  options, required options, short aliases, validation, duplicate-option failures,
  dash-prefixed token behavior, unexpected positional behavior, and all-failures
  startup validation (FR-014).
- Run `cargo test -p webe_args --doc` to confirm documentation examples compile and
  pass, keeping docs in sync with behavior.

## Facade check (C15)

```bash
cargo build   # default features include `args`
```

Confirm the crate is reachable via `webe::args` from the workspace facade
(`src/lib.rs`).

## Success signal

The feature is validated when: the build/lint/test commands above succeed,
scenarios 1–17 are covered by passing tests, doctests pass, and `webe::args`
resolves under default features.
